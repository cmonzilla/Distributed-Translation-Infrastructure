7 ## Introduction
8 This is a fork project from the Automated-Translation-Tries project, made as a test exercise for automated machine translation (aiming at automated translation of text in different languages).
9 
10 For machine translation it is important to estimate and compare the fluency of different possible translation outputs for the same source (i.e., foreign) sentence. This is commonly achieved by using a language model, which measures the probability of a string (which is commonly a sentence). Since entire sentences are unlikely to occur more than once, this is often approximated by using sliding windows of words (n-grams) occurring in some training data.
11 
12 ### Background
13 An <i>n-gram</i> refers to a continuous sequence of n tokens. For instance, given the following sentence: our neighbor , who moved in recently , came by . If n = 3, then the possible n-grams of
14 this sentence include:
15 <code>
16 "our neighbor ,"
17 "neighbor , who"
18 ", who moved"
19 ...
20 ", came by"
21 "came by ."
22 </code>
23 
24 Note that punctuation marks such as comma and full stop are treated just like any ‘real’ word and that all words are lower cased.

118 ##Building the project
119 Building this project requires gcc version 4.9.1 and higher. The project can be build in two ways:
120 
121 + From the Netbeans environment by running Build in the IDE
122 + From the Linux command-line console
123  - Open the console
124  - Navigate to the project folder
125  - Run <i>"make all"</i>
126  - The binary will be generated and placed into <i>./dist/[type]_[platform]/</i> folder
127  - The name of the executable is <i>back-off-language-model-smt</i>
128 
129 In order to clean the project from the command line run <i>"make clean"</i>.
130 
131 For the sake of simplicity and speed building on Centos platform is also possible by using make_*_centos.sh scripts. There are four of suck scripts available:
132 
133 * make_debug_centos.sh - allows to build the debug configuration: ready to be used with valgrind and gdb.
134 * make_profile_centos.sh - allows to build the profiling configuration: ready to be used for gathering performance information for gprof: produce the gmon.out file.
135 * make_release_centos.sh - allows to build the release configuration: the version build for performance with platform specific optimization's.
136 * make_centos.sh - allows to build all the above configurations at once.
137 
138 One can limit the debug-level printing of the code by changing the value of the <i>LOGER_MAX_LEVEL</i> constant in the <i>./inc/Configuration.hpp</i>. The possible range of values, with increasing logging level is: ERROR, WARNING, USAGE, RESULT, INFO, INFO1, INFO2, INFO3, DEBUG, DEBUG1, DEBUG2, DEBUG3, DEBUG4. It is also possible to vary the information level output by the program during its execution by specifying the command line flag, see the next section.

183 ##Implementation Details
184 
185 In this section we mention a few implementation details, for more details see the source code documentation. At present the documentation is done in the Java-Doc style that is successfully accepted by Doxygen with the Doxygen option <i>JAVADOC_AUTOBRIEF</i> set to <i>YES</i>. The generated documentation is located in the <big>./doxygen/</big> folder of the project.
186 
187 The code contains the following important source files:
188 
189 * <big>main.cpp</big> - contains the entry point of the program
190 * <big>Executor.cpp</big> - contains some utility functions including the one reading the test document and performing the queries on a filled in Trie instance.
191 * <big>ARPATrieBuilder.hpp / ARPATrieBuilder.cpp</big> - contains the class responsible for reading the ARPA file format and building up the trie model using the ARPAGramBuilder.
192 * <big>TrieDriver.hpp</big> - is the driver for all trie implementations - allows to execute queries to the tries.
193 * <big>LayeredTrieDriver.hpp</big> - is a wrapper driver for all the layered trie implementations - allows to retrieve N-gram probabilities and back-off weights.
194 * <big>C2DHashMapTrie.hpp / C2DHashMapTrie.cpp</big> - contains the Context-to-Data mapping trie implementation based on unordered_map.
195 * <big>C2DMapArrayTrie.hpp / C2DMapArrayTrie.cpp</big> - contains the Context-to-Data mapping trie implementation based on unordered_map and ordered arrays.
196 * <big>C2WOrderedArrayTrie.hpp / C2WOrderedArrayTrie.cpp</big> - contains the Context-to-Word mapping trie implementation based on ordered arrays.
197 * <big>G2DHashMapTrie.hpp / G2DHashMapTrie.cpp</big> - contains the M-Gram-to-Data mapping trie implementation based on self-made hash maps.
198 * <big>W2CHybridMemoryTrie.hpp / W2CHybridMemoryTrie.cpp</big> - contains the Word-to-Context mapping trie implementation based on unordered_map and ordered arrays.
199 * <big>W2COrderedArrayTrie.hpp / W2COrderedArrayTrie.cpp</big> - contains the Word-to-Context mapping trie implementation based on ordered arrays.
200 * <big>Configuration.hpp</big> - contains configuration parameter for the word index and trie and memory management entities.
201 * <big>Exceptions.hpp</big> - stores the implementations of the used exception classes.
202 * <big>HashingUtils.hpp</big> - stores the hashing utility functions.
203 * <big> ARPAGramBuilder.hpp / ARPAGramBuilder.cpp</big> - contains the class responsible for building n-grams from a line of text and storing it into Trie.
204 * <big>StatisticsMonitor.hpp / StatisticsMonitor.cpp</big> - contains a class responsible for gathering memory and CPU usage statistics
205 * <big>Logger.hpp/Logger.cpp</big> - contains a basic logging facility class

207 ##ToDo
208 * <big> C2DHashMapTrie.hpp / C2DHashMapTrie.cpp </big> - the current implementation is potentially error prone to hash collisions in case of context id overflows. Overflows were not observed on the tries of up to 20 Gb but a more thorough testing must be needed and perhaps the collision detection must be always on for this trie.
209 * <big>Tries</big> - It is possible to introduce more templating into the tries, e.g. the gram-level-based templating. It must improve performance as many checks can be resolved compile-time.
210 * <big>G2DHashMapTrie.hpp / G2DHashMapTrie.cpp</big> - This trie is very performance efficient but its memory consumption is at present sub optimal. It needs a significant re-work in the way data is stored.
211 * <big>Thread safety</big> - Not all the code is thread safe. Tries are to be reviewed for using class data members during filling in the tries or querying. One can just make the entire trie interface synchronized but this is sub-optimal therefore the idea is, when querying, to use the shared class members only for reading and all the temporary storage data is to be allocated and passed through the call stack by reference. This is, for the most, already so but requires and extra check.
212 * <big>Testing</big> - the testing done with this code was limited. Potentially the Trie code, and the rest, still contains error. So it is recommended to add unit and functional tests for this project
213 * <big>Code</big> - in some places more of the old style C functions are used, which might have good equivalent in C++. Also, the naming convention is not always ideally followed. The using of Templates in the code might be to complex, although potentially gives some performance and genericity advantages.
